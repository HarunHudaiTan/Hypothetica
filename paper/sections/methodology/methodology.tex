\section{Methodology}
\label{sec:methodology}

Validating a new idea before committing to it requires manually searching through thousands of academic publications, reading abstracts, downloading relevant resources, and forming a judgment about whether the core concept, methodology, and claimed contributions are different from prior work. For a single idea, this process can take days or weeks, and it is time consuming. A person working alone cannot realistically search relevant literature across multiple keyword formulations, related domains, and methodological variations at the same time. The result is that many projects begin with incomplete validation, discovering conflicts with existing work only after significant time and resources have already been invested. This challenge is not limited to academic researchers. Students proposing thesis topics, engineers evaluating whether a technical approach has already been patented or published, and innovators assessing the novelty of a product concept all face the same problem: there is too much existing work to go through. Because of this overwhelming volume of information, carefully verifying everything by hand becomes almost impossible in practice.

Hypothetica addresses this problem by automating both the search and the evaluation stages of the originality validation process. Rather than requiring a user to manually search resources, read them, and form a subjective judgment, the system handles the entire process computationally. It works by generating several different search queries that capture the meaning of the submitted idea from multiple angles. Using these queries, the system gathers a wide range of potentially related resources from academic literature using OpenAlex and industrial prior art using Google Patents. It then filters and ranks these resources based on how relevant they are to the idea. After identifying the most relevant ones, the system analyzes their content and evaluates each paper in relation to the idea across four different dimensions. The final output is not just a single, unclear similarity score. Instead, the system provides a structured and evidence-based originality assessment. It directly connects specific sentences from the user's idea with matching passages found in existing papers, making it easy to see exactly where similarities occur and how meaningful those similarities are. This way, users can understand the relationship between their idea and prior work rather than relying on an unclear numerical score.

The system is built on the idea that conceptual similarity and textual similarity are not the same thing. Two ideas might describe the exact same approach while using completely different words, and at the same time a truly novel idea might still share common terminology with existing research simply because it belongs to the same field. Traditional text-matching tools struggle with this difference, since they mostly rely on keywords or surface-level similarity.

Hypothetica approaches the problem differently by working at the semantic level. Instead of focusing only on matching words, it uses dense vector embeddings and large language model reasoning to understand the underlying meaning of ideas. By analyzing concepts rather than just vocabulary, the system can detect deeper similarities between works even when they are expressed in very different ways. As a result, it can reveal hidden similarities that simple keyword searches or standard similarity tools would likely miss.

\subsection{System Overview}

Hypothetica is structured as a multi-phase pipeline that takes a natural language research idea as input and produces a structured originality assessment as output. The pipeline begins with a User Interaction Phase that turns the raw submission into an enriched, well-defined idea description, and continues through a Retrieval Phase that gathers candidate prior art from the user's selected evidence source. A Candidate Selection Phase then narrows this pool down to a small, high-quality shortlist through a combination of vector-based filtering, cross-encoder reranking, and an LLM-based selection step. Once the shortlist is fixed, the Parsing Phase converts each selected resource into structured, searchable text and indexes it in a vector store. The two scoring phases that follow are where the originality verdict is produced: the Candidate Originality Scoring phase analyzes each shortlisted resource independently against the user's idea and produces per-criterion similarity scores together with sentence-level overlap evidence, and the Idea Originality Scoring phase aggregates these per-resource results into a single global originality score, a labeled breakdown across the four originality criteria, and a sentence-level annotation of the user's idea. The final output, presented to the user through the interface, includes the global originality score, a color-coded breakdown of the idea where each sentence is labeled as having high, moderate, or low originality, and expandable evidence panels that show the specific passages that drove each label.

The phases communicate through structured data passed from one stage to the next, and the boundaries between them are deliberate: each phase has a single, well-defined responsibility, which makes the pipeline easier to extend with new evidence sources, new scoring criteria, or new user-facing visualizations without disturbing the rest of the system. The remainder of this section describes each phase in detail.

\subsection{User Interaction Phase}

\paperfigureblock{fig_user_interaction.png}{User Interaction Diagram.}{fig:user_interaction}

The pipeline begins with the User Interaction Phase, which transforms a raw user submission into a structured input that the rest of the system can operate on. As shown in Fig.~\ref{fig:user_interaction}, this phase takes a User Request Payload and produces an Enriched Idea that is passed forward to the Retrieval Phase.

The first step is an Adapter Availability Check. Before any analysis begins, the system verifies that the adapter chosen by the user, such as OpenAlex, Google Patents, or GitHub, is configured and accessible. If the selected adapter is unavailable, the request is rejected. This early validation prevents wasted computation and gives the user immediate feedback when an evidence source cannot be reached.

Once the adapter is confirmed to be available, the Follow-Up Questions step is triggered. A dedicated agent reads the user's research idea and generates three clarifying questions. These questions are designed to cover aspects such as the specific research problem being addressed, the proposed method or approach, and the parts of the idea the user considers novel. The purpose of this step is to surface information that researchers often leave implicit in a short idea description but that is critical for accurate originality assessment. Without this clarification, vague ideas tend to lead to poor retrieval and shallow per-paper analysis later in the pipeline.

The user's responses to these questions are then passed to the Idea Enrichment step. Here, the original idea and the question-answer pairs are combined into a single, structured text block that preserves both the user's initial idea and the additional details provided through the clarifications. This combined text is the Enriched Idea that drives later stages of the pipeline. From this point, retrieval queries, candidate selection, per-paper scoring, and global originality scoring all operate on the enriched description rather than on the raw input.

\subsection{Retrieval Phase}

\paperfigureblock{fig_retrieval.png}{Retrieval Phase Diagram.}{fig:retrieval}

Once the Enriched Idea Context is produced, the pipeline enters the Retrieval Phase, which is responsible for retrieving a large pool of candidate resources from the selected adapter. As shown in Fig.~\ref{fig:retrieval}, this phase takes the Retrieval Input consisting of the Enriched Idea and the Selected Adapter chosen earlier by the user and outputs N Unique Candidates that are forwarded to the Candidate Selection Phase. The main challenge at this stage is balancing recall and precision. If the search is too narrow, it risks missing important prior work altogether. On the other hand, if it is too broad, it brings back a large number of documents, many of which are not relevant. For this reason, the Retrieval Phase is designed to focus more on recall, while precision is improved later during the filtering steps.

The process starts with Query Variant Generation. Instead of relying on a single query based on the user's idea, a Query Variant Agent generates four or five different queries that express the same idea from multiple angles. Each one has a specific role. One is a simple, natural-language description of the idea. Two others are written in a more academic style, similar to how papers are titled or described -- one focusing on the method, the other on the application area. There is also a query built from synonyms and abbreviations, using OR operators to capture different ways the same concept might be written.

The agent also adapts to the data source. For example, if the adapter is GitHub, it shifts toward more implementation-focused wording that matches repository names and README files. If the adapter is OpenAlex, it instead produces broader, concept-level queries that work better with its full-text search. In the end, this step produces a list of query variants that represent the same idea in several complementary ways.

The Variant Queries List is then sent, one query at a time, to the Adapter Search step. The selected adapter -- OpenAlex, Google Patents, or GitHub -- is invoked through a unified adapter interface that hides the differences between the underlying APIs. Each adapter is responsible for issuing the query to its corresponding service, parsing the source-specific response, and converting the raw results into a standardized Resource representation that the rest of the pipeline can operate on. For each query variant, the adapter retrieves up to a configurable number of candidates. Operating through this adapter abstraction makes it straightforward to add new evidence sources in the future without changing the rest of the pipeline.

After all the variant searches finish, the Deduplication step merges their results into a single pool. Since each variant describes the same idea from a slightly different angle, the same resource often shows up in more than one result set, and the system needs to make sure it is only carried forward once. To handle this, deduplication relies on stable identifiers like the OpenAlex Work ID, the patent ID, or the repository's full name, and falls back to DOI when one is available. Anything that survives this step is given a fresh internal ID so the rest of the pipeline can refer to it consistently.

The final output of the Retrieval Phase is a set of N Unique Candidates, where N typically ranges from a few dozen to several hundred depending on the chosen adapter and configuration. This pool is intentionally larger than what would be analyzed in detail, since the goal of this phase is to maximize the chance that any prior work that genuinely overlaps with the user's idea is included. Reducing this pool to a small, high-quality set is the responsibility of the next stage, the Candidate Selection Phase.

\subsection{Candidate Selection Phase}

\paperfigureblock{fig_candidate_selection.png}{Candidate Selection Diagram.}{fig:candidate_selection}

The Candidate Selection Phase is responsible for tightening this pool down to a small, high-quality set of resources that will be used for detailed analysis. As shown in Fig.~\ref{fig:candidate_selection}, the input to this phase is the Candidate Selection Input, consisting of N Candidates from the Retrieval Phase together with the Enriched Idea, and the output is a set of Top K Selected Candidates that will be passed to the Parsing Phase. The phase is structured as a three-step process that combines vector-based filtering, a more accurate but heavier reranking step, and finally a Large Language Model for judgment.

The first stage is Embedding Generation. Each candidate's title and abstract or, in the case of GitHub, the repository description and README are embedded into a dense vector representation using a pretrained text embedding model. Once embeddings are available, the Semantic Similarity Ranking step computes the cosine similarity between the Enriched Idea vector and each candidate vector, and keeps the most similar ones according to a configurable cutoff. This is where most of the noise in the pool gets filtered out: candidates that share keywords with the idea but are conceptually unrelated tend to fall away, while candidates that talk about the same underlying problem in different words tend to surface. At this stage the system keeps the candidate set wide rather than aggressive, because the goal is to feed a high-recall input into the more precise reranker.

The second stage is Cross Encoder Reranking. Unlike the embedding step, which encodes the query and each document independently and only compares them at the end, a cross encoder takes the query and a single document together and produces a similarity score in one pass. This makes it more accurate, because the model can attend to the interaction between specific words in the query and specific words in the document, but also considerably more expensive, which is why it only runs on the top candidates produced by the embedding step. The output of this stage is the Scored Search Results: a top-ranked shortlist whose size is configurable by the user and defaults to 20, ready to be passed to a language model for the final selection.

The final stage is the Relevance Selection Agent, which acts as an LLM-as-a-judge over the reranked shortlist. The agent receives the Enriched Idea together with the title, abstract, and metadata of every candidate that survived the cross-encoder step, and is asked to select the resources that are most relevant for a deeper originality analysis. The agent does not rely only on similarity scores; it is prompted to consider conceptual alignment as well, including whether the candidate addresses a similar problem, uses related methods, or makes comparable contributions to those of the user's idea. This judgment catches cases where two candidates have nearly identical similarity scores but differ in how directly they engage with the user's idea, and it is also where adapter-aware reasoning is applied, since the agent receives slightly different selection criteria for repositories than for academic resources or patents.

The output of the Candidate Selection Phase is a set of Top K Selected Candidates, where K is configurable by the user and defaults to five. These resources represent the final resource list that will be analyzed in detail in the rest of the pipeline. By the time a resource reaches this point, it has passed three filters -- vector similarity, cross-encoder reranking, and language model judgment -- which together provide a much stronger guarantee of relevance than any single retrieval signal could offer on its own.

\subsection{Parsing Phase}

\paperfigureblock{fig_parsing.png}{Parsing Diagram.}{fig:parsing}

Before any per-candidate analysis can take place, each of the selected resources needs to be turned into structured, searchable text. This is the responsibility of the Parsing Phase, shown in Fig.~\ref{fig:parsing}. The phase takes the Top-K Candidates produced by the Candidate Selection Phase and ends with a populated ChromaDB Vector Store containing the indexed chunks of every candidate, ready for retrieval during the scoring phases.

The phase begins with Parallel Document Fetching. Each candidate's full content is downloaded concurrently. PDFs are retrieved for OpenAlex, the Google Patents API is queried directly for patent descriptions, and the README content already collected during retrieval is reused for GitHub repositories. The fetched content is then converted into a Markdown representation so that the rest of the pipeline can treat resources from different adapters in the same way.

Once a uniform document is available, the Extract Section Content step parses out the structural skeleton of each resource. Headings are identified through pattern matching on the Markdown, and the text under each heading is associated with that heading as its section content. Sections that contribute little to originality assessment, such as references, acknowledgments, and patent-specific legal sections, are skipped at this stage so that they do not pollute the downstream representation.

The remaining steps are repeated for each candidate. The Document Chunking Processor splits each section into smaller chunks, with target sizes chosen to fit the context windows of both the embedding model and the language model. The Vector Store Ingestion step then embeds these chunks and writes them into the ChromaDB Vector Store along with metadata. The output of the Parsing Phase is the populated ChromaDB Vector Store, indexed and ready to support retrieval during the Candidate Originality Scoring and Idea Originality Scoring phases.

\subsection{Candidate Originality Scoring}

\paperfigureblock{fig_candidate_scoring.png}{Candidate Originality Scoring Diagram.}{fig:candidate_scoring}

The Candidate Originality Scoring phase is where each candidate is independently compared against the user's idea to produce a structured similarity profile. As shown in Fig.~\ref{fig:candidate_scoring}, the input to this phase is the Enriched Idea together with the processed candidates and the ChromaDB index, and the output is a per-candidate Candidate Originality Scoring Result containing a similarity score, criteria-level scores, sentence-level analysis, matched sections, and a justification for each score. The phase loops over every candidate and treats each one as a separate scoring task, so that one strongly overlapping resource cannot be averaged away by a group of weakly related ones.

For each candidate, the system first builds a focused textual representation of the resource. The Extracted Fixed Sections step pulls out the parts of the document that are almost always relevant for originality assessment -- the abstract, introduction, and conclusion -- directly from the parsed Markdown. In parallel, the Contextual RAG Retrieval step queries the ChromaDB vector store with the Enriched Idea and retrieves the chunks from the same candidate that are most semantically aligned with it. Combining a fixed structural skeleton with retrieval-driven context lets the language model see both the standard parts of the resource and the parts that specifically discuss similar problems or methods, without overwhelming the prompt with the entire document.

This combined context is then passed to two parallel evaluation tracks. The first track is the Multi-Criterion LLM Evaluation, which scores the candidate against the Enriched Idea on four originality dimensions: Problem Similarity, Method Similarity, Domain Similarity, and Contribution Similarity. Each criterion is scored in its own LLM call, with a criterion-specific rubric and explicit definitions for every level on a 5-point Likert scale. Decoupling the criteria into separate calls is a deliberate design choice. Asking a single prompt to evaluate all four dimensions simultaneously tends to cause criterion bleeding, where the model conflates dimensions and produces inconsistent scores; isolating each criterion forces the model to focus on the corresponding rubric and produces noticeably more stable judgments. Each call also returns a short justification grounded in specific content from the candidate, which is later surfaced to the user as evidence.

The second track is the Sentence-Level Overlap Analysis, which examines the user's idea at a finer granularity. Instead of evaluating the idea as a whole, this step scores every individual sentence of the Enriched Idea against the candidate, again using the same four criteria and the same 5-point Likert scale. For each sentence, the LLM also identifies the specific passages in the candidate that drove the overlap, recording the section heading, a short evidence snippet, and a brief reason. To keep sentence-level scores from drifting away from the candidate-level judgment, sentence-level criteria scores are filtered against the candidate-level scores: a sentence cannot be flagged as more overlapping on a given criterion than the candidate-level rubric allows. This consistency anchor prevents isolated sentences from inflating the overall picture.

The Likert scores produced by the two tracks are then mapped to floats in the $[0, 1]$ range, where 1 corresponds to near-complete overlap and 0 corresponds to no meaningful overlap. The Similarity Score Calculation step combines the four candidate-level criteria scores into a single per-candidate similarity score using a weighted average:

\begin{equation}
	\text{sim}_{\text{candidate}} = w_p \cdot S_{\text{problem}} + w_m \cdot S_{\text{method}} + w_d \cdot S_{\text{domain}} + w_c \cdot S_{\text{contribution}}
	\label{eq:sim_candidate}
\end{equation}

The output of this phase, for each candidate, is a Candidate Originality Scoring Result containing the per-candidate similarity score, the four criteria scores together with their justifications, the sentence-level overlap analysis, and the matched evidence sections retrieved from the candidate. These results are the structured input to the next phase, where they are aggregated into a single global originality score for the user's idea.

\subsection{Idea Originality Scoring}

\paperfigureblock{fig_idea_scoring.png}{Idea Originality Scoring Chart.}{fig:idea_scoring}

The Candidate Originality Scoring phase produces an independent similarity profile for every Top-K candidate. The Idea Originality Scoring phase takes this collection of per-candidate results and turns it into a single, user-facing assessment of how original the user's idea is. As shown in Fig.~\ref{fig:idea_scoring}, the input to this phase is the set of scoring results for every candidate, and the output is the Idea Originality Scoring Result that drives the entire results page in the user interface. The phase has three parallel branches -- Aggregated Criteria, Global Similarity Computation, and Sentence Annotations -- followed by a final originality score derived from the global similarity.

The Global Similarity Computation branch determines the final originality verdict. Each candidate already carries a per-candidate similarity score $\text{sim}_i$ produced in the previous phase, where $i$ indexes the $N$ selected candidates. A naive aggregation that simply averages these scores would dilute a strong overlap signal whenever the retrieval pipeline returned a few weakly related candidates alongside a genuinely close match. To preserve that signal while still being robust to noise, the global similarity is computed as a blend of the maximum and the average of the per-candidate scores:

\begin{equation}
	\text{sim}_{\text{global}} = w_{\max} \cdot \max_{i}(\text{sim}_{i}) + \frac{w_{\text{mean}}}{N} \sum_{i=1}^{N} \text{sim}_{i}
	\label{eq:sim_global}
\end{equation}

The two weights $w_{\max}$ and $w_{\text{mean}}$ sum to one and are configurable. Setting $w_{\max}$ higher than $w_{\text{mean}}$ means that a single candidate with high overlap dominates the result, which matches the intuition that one strongly overlapping prior work is enough to challenge the originality of an idea. At the same time, including the average term dampens the effect of a single overscored outlier, so an isolated noisy candidate cannot single-handedly collapse the score.

The global similarity is then converted to a final originality score on a 0--100 scale through a power curve rather than a linear mapping:

\begin{equation}
	\text{originality} = \left(1 - \text{sim}_{\text{global}}^{p}\right) \cdot 100
	\label{eq:originality}
\end{equation}

The exponent $p$ is greater than one, which makes the curve concave with respect to similarity: high similarity values are penalized more strongly than low ones. This deliberately pushes borderline cases toward the lower end of the originality range, since in an originality assessment context it is safer to flag a moderately overlapping idea than to give it the benefit of the doubt. The resulting integer score is then mapped to one of three labels -- high, medium, or low originality -- using configurable thresholds, and these labels drive the color coding shown in the user interface.

In parallel with the global computation, the Criteria Breakdown branch produces a single value per criterion across all candidates for display purposes. For each criterion separately, letting $c_i$ denote the score of candidate $i$ on that criterion, the system combines the per-candidate scores using the same blend of maximum and mean as the global similarity:

\begin{equation}
	S_{\text{breakdown}} = w_{\max} \cdot \max_{i}(c_i) + \frac{1 - w_{\max}}{N} \cdot \sum_{i=1}^{N} c_i
	\label{eq:breakdown}
\end{equation}

so that the most alarming candidate on a given criterion has the strongest influence on the displayed value while the others still contribute. Using the same $w_{\max}$ weight as the global similarity keeps the breakdown view consistent with the blend that drives the final score. These breakdown scores are not used in the final originality score; they exist to give users a quick visual breakdown of which dimensions of their idea overlap most with prior work.

The Sentence Annotations branch operates at a finer granularity. For every sentence in the Enriched Idea, the system collects the sentence-level criterion scores assigned to it by every candidate during the previous phase, takes the maximum per criterion across candidates, and converts that maximum back into a Likert level. Each criterion is then labeled red, yellow, or green based on configurable Likert thresholds, and the overall sentence label is the worst label across its criteria. Each labeled sentence is also linked to the strongest matching evidence section retrieved from the candidate that drove the score, so that users can inspect exactly which passage in which candidate caused a given annotation. Sentence annotations are display artifacts; they do not affect the final originality score.

The three branches are then combined into the Idea Originality Scoring Result, which contains the final originality score and label, the aggregated criteria values, the per-sentence annotations with their linked evidence, and a short natural-language summary generated by a dedicated Summary Agent. The summary takes the global score, the aggregated criteria, and the distribution of sentence labels as input, and produces a one-to-two-sentence explanation that highlights the main areas of overlap and offers an actionable recommendation. This final, structured object is what the system passes to the user interface to render the originality report.
